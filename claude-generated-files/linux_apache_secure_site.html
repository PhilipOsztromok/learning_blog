🔒 Secure the Site and Control Access

A publicly accessible web server needs to be hardened before it is truly production-ready. This lesson covers the full security stack: encrypting traffic with SSL/TLS, obtaining certificates from a trusted authority, configuring HTTPS in Apache, and controlling who can access your content using .htaccess rules for both authentication and URL rewriting.

Platform note: Commands and file paths differ between Debian-based systems (Raspberry Pi / Ubuntu) and Red Hat-based systems (RHEL / Rocky / Fedora). Platform-specific blocks are clearly labelled throughout this lesson.

1 — Secure Access with SSL

SSL (Secure Sockets Layer) and its modern successor TLS (Transport Layer Security) encrypt the connection between a browser and your web server, protecting data in transit from eavesdropping and tampering. In practice, the two terms are used interchangeably — when people say "SSL" they usually mean the modern TLS protocol.

Without HTTPS, all data sent between the browser and your server — including passwords, form submissions, and session tokens — travels as plain text across the network and can be intercepted by anyone on the same network path.

Encryption
Data in Transit
TLS encrypts all data between the browser and server. Even if traffic is intercepted, it cannot be read without the private key. Passwords, cookies, and form data are all protected.
Authentication
Certificate Identity
A TLS certificate proves that the server is who it claims to be. Browsers verify this against a list of trusted Certificate Authorities (CAs). This prevents man-in-the-middle attacks.
Integrity
Tamper Detection
TLS includes a message authentication code (MAC) that detects if any data was altered in transit. The browser will reject a modified response before rendering it.
Apache Module
mod_ssl
Apache implements TLS through mod_ssl. This module must be enabled and configured before Apache can serve HTTPS traffic on port 443. It handles the TLS handshake and certificate presentation.

Enable mod_ssl

🐧 Debian / Raspberry Pi
# Enable the SSL module sudo a2enmod ssl # Enable the default SSL site (optional) sudo a2ensite default-ssl # Reload Apache sudo systemctl reload apache2 # Confirm mod_ssl is loaded apache2ctl -M | grep ssl ssl_module (shared)
🎩 Red Hat / Rocky / Fedora
# Install mod_ssl (includes the module) sudo dnf install -y mod_ssl # mod_ssl auto-loads via: /etc/httpd/conf.modules.d/00-ssl.conf # Reload Apache sudo systemctl reload httpd # Confirm mod_ssl is loaded httpd -M | grep ssl ssl_module (shared)
On Red Hat, installing mod_ssl also creates a default self-signed certificate at /etc/pki/tls/.
💡 HTTPS is now expected, not optional. Modern browsers mark HTTP sites as "Not Secure". Search engines rank HTTPS sites higher. Many browser APIs (geolocation, service workers, camera) are restricted to HTTPS-only contexts. There is no good reason to run a public-facing site without HTTPS in 2025.

2 — Create a Self-Signed Certificate

A self-signed certificate is one that you generate yourself, without involving a Certificate Authority. The certificate is signed with its own private key rather than by a trusted third party. Browsers will display a security warning because they cannot verify the issuer — but the traffic is still fully encrypted.

Use Case
Development & Internal
Self-signed certificates are appropriate for local development servers, internal tools, home lab projects, and testing TLS configuration before obtaining a real certificate. Not suitable for public-facing production sites.
Browser Behaviour
Security Warning
Browsers will display a "Your connection is not private" warning and require the user to manually accept the risk. This is expected and normal — the certificate is not trusted by the browser's built-in CA list.

Generate a Self-Signed Certificate with OpenSSL

🐧 Both Platforms — OpenSSL command
# Generate a private key and self-signed certificate in one command # This creates a certificate valid for 365 days using RSA 2048-bit key sudo openssl req -x509 -nodes -days 365 -newkey rsa:2048 \ -keyout /etc/ssl/private/mysite.key \ -out /etc/ssl/certs/mysite.crt # You will be prompted for certificate details: Country Name (2 letter code) [AU]: GB State or Province Name: England Locality Name: London Organization Name: My Company Organizational Unit Name: IT Common Name (e.g. server FQDN): mysite.com Email Address: admin@mysite.com
The Common Name must match the domain name you will use to access the server, otherwise browsers will show an additional hostname mismatch warning.

OpenSSL Flag Breakdown

FlagMeaning
req -x509Create a self-signed certificate (skip the CA signing step)
-nodesNo DES — do not encrypt the private key with a passphrase (Apache needs to read it on startup without prompting)
-days 365Certificate validity period in days
-newkey rsa:2048Generate a new 2048-bit RSA private key at the same time
-keyoutPath to write the private key file
-outPath to write the certificate file

Secure the Private Key File

🐧 Both Platforms
# The private key must be readable only by root sudo chmod 600 /etc/ssl/private/mysite.key sudo chown root:root /etc/ssl/private/mysite.key # Verify permissions ls -la /etc/ssl/private/mysite.key -rw------- 1 root root 1704 Aug 15 10:00 mysite.key # Verify the certificate details openssl x509 -in /etc/ssl/certs/mysite.crt -text -noout | head -20
⚠️ Never share your private key. The .key file must remain on the server and be readable only by root. If it is ever compromised, anyone who has it can impersonate your server. Revoke and replace the certificate immediately if a key is exposed.

3 — Use a Certificate from a Certificate Authority (CA)

For any public-facing website, you need a certificate signed by a Certificate Authority (CA) that browsers already trust. Let's Encrypt is a free, automated, and open CA that has become the standard choice for most websites. The Certbot tool handles obtaining, installing, and automatically renewing Let's Encrypt certificates.

Certificate Authority
What a CA Does
A CA verifies that you control the domain you are requesting a certificate for, then signs the certificate with its own private key. Browsers ship with a list of trusted CAs, so any certificate they sign is automatically trusted.
Let's Encrypt
Free & Automated
Let's Encrypt issues certificates at no cost and automates the domain verification (ACME protocol). Certificates expire after 90 days but Certbot handles automatic renewal so you rarely need to think about it.

Install Certbot

🐧 Debian / Raspberry Pi / Ubuntu
# Install Certbot and the Apache plugin sudo apt update sudo apt install -y certbot python3-certbot-apache # Verify installation certbot --version certbot 2.x.x
🎩 Red Hat / Rocky / Fedora
# Enable EPEL repository first (if not already enabled) sudo dnf install -y epel-release # Install Certbot and Apache plugin sudo dnf install -y certbot python3-certbot-apache # Verify installation certbot --version

Obtain a Certificate

🐧 Both Platforms — Certbot with Apache plugin
# Obtain a certificate AND automatically configure Apache # Replace mysite.com with your actual domain sudo certbot --apache -d mysite.com -d www.mysite.com # Certbot will: # 1. Verify you control the domain (HTTP challenge) # 2. Download the signed certificate # 3. Update your Apache virtual host config automatically # 4. Set up a redirect from HTTP to HTTPS # Certificates are stored at: /etc/letsencrypt/live/mysite.com/fullchain.pem # certificate + chain /etc/letsencrypt/live/mysite.com/privkey.pem # private key /etc/letsencrypt/live/mysite.com/cert.pem # certificate only /etc/letsencrypt/live/mysite.com/chain.pem # CA chain only

Test and Automate Renewal

🐧 Both Platforms
# Test that renewal would succeed (dry run — no changes made) sudo certbot renew --dry-run # Certbot installs a systemd timer (or cron job) automatically # to renew certificates before they expire. Check it: sudo systemctl status certbot.timer # Or check the cron job: cat /etc/cron.d/certbot # Manually trigger renewal (if within 30 days of expiry) sudo certbot renew # Check certificate expiry date sudo certbot certificates Found the following certs: Certificate Name: mysite.com Domains: mysite.com www.mysite.com Expiry Date: 2025-11-14 (VALID: 89 days)
Domain requirement: Let's Encrypt certificates require a publicly accessible domain name that resolves to your server's IP address. They cannot be issued for bare IP addresses, localhost, or internal hostnames. For those cases, use a self-signed certificate instead.

4 — Add an SSL Certificate to Your Site

Once you have a certificate (self-signed or from a CA), you need to configure Apache to use it for your virtual host. This involves creating a VirtualHost block on port 443, pointing Apache to your certificate files, and redirecting HTTP traffic to HTTPS.

HTTPS Virtual Host Configuration

Complete SSL VirtualHost — Both Platforms
This configuration serves HTTPS on port 443. Use the correct certificate paths for your setup — paths differ between self-signed and Let's Encrypt certificates.
# Port 443 — HTTPS virtual host <VirtualHost *:443> ServerName mysite.com ServerAlias www.mysite.com DocumentRoot /var/www/mysite # Enable SSL for this virtual host SSLEngine on # --- Self-signed certificate paths --- SSLCertificateFile /etc/ssl/certs/mysite.crt SSLCertificateKeyFile /etc/ssl/private/mysite.key # --- Let's Encrypt certificate paths --- # SSLCertificateFile /etc/letsencrypt/live/mysite.com/fullchain.pem # SSLCertificateKeyFile /etc/letsencrypt/live/mysite.com/privkey.pem <Directory /var/www/mysite> AllowOverride All Require all granted </Directory> ErrorLog ${APACHE_LOG_DIR}/mysite-ssl-error.log CustomLog ${APACHE_LOG_DIR}/mysite-ssl-access.log combined </VirtualHost>

Redirect HTTP to HTTPS

Force all HTTP traffic to HTTPS
Add a second VirtualHost on port 80 that permanently redirects all traffic to the HTTPS version. Place this in the same config file, above the port 443 block.
# Port 80 — redirect all HTTP to HTTPS <VirtualHost *:80> ServerName mysite.com ServerAlias www.mysite.com # 301 = permanent redirect Redirect permanent / https://mysite.com/ </VirtualHost>

Enable and Activate the Configuration

🐧 Debian / Raspberry Pi
# Save config to sites-available sudo nano /etc/apache2/sites-available/mysite-ssl.conf # Ensure mod_ssl is enabled sudo a2enmod ssl # Enable the site sudo a2ensite mysite-ssl.conf # Test and reload sudo apache2ctl configtest sudo systemctl reload apache2 # Verify HTTPS is working curl -Ik https://mysite.com HTTP/2 200
🎩 Red Hat / Rocky
# Save config to conf.d sudo nano /etc/httpd/conf.d/mysite-ssl.conf # Ensure mod_ssl is installed sudo dnf install -y mod_ssl # Open port 443 in firewall sudo firewall-cmd --permanent --add-service=https sudo firewall-cmd --reload # Test and reload sudo apachectl configtest sudo systemctl reload httpd # SELinux: allow Apache to read cert files sudo restorecon -Rv /etc/ssl/
💡 The -k flag in curl skips certificate verification — useful for testing self-signed certificates locally. Never use it in production scripts, as it defeats the purpose of HTTPS.

5 — Protect a Site with a .htaccess File

A .htaccess file (hypertext access) is a per-directory configuration file that Apache reads on every request. It lets you add access controls, password protection, and redirects without editing the main Apache configuration. To use .htaccess files, AllowOverride All must be set in the Directory block for that path.

Password Protection with Basic Authentication

HTTP Basic Authentication challenges the user with a username/password dialog before serving any content from the protected directory. Credentials are stored in a flat .htpasswd file using a hashed format.

🐧 Both Platforms — Create the .htpasswd file
# Install apache2-utils (Debian) / httpd-tools (Red Hat) for htpasswd sudo apt install apache2-utils # Debian sudo dnf install httpd-tools # Red Hat # Create a new .htpasswd file with first user (-c creates the file) sudo htpasswd -c /etc/apache2/.htpasswd admin New password: •••••••• Re-type new password: •••••••• Adding password for user admin # Add additional users (omit -c or it will overwrite the file) sudo htpasswd /etc/apache2/.htpasswd philip # View the file (passwords are hashed — not plain text) cat /etc/apache2/.htpasswd admin:$apr1$rBGHs...$XQm4... philip:$apr1$tYKmn...$nWp5... # Delete a user from the file sudo htpasswd -D /etc/apache2/.htpasswd philip
.htaccess — Password Protect a Directory
Place this .htaccess file inside the directory you want to protect (e.g. /var/www/mysite/admin/.htaccess).
# Set the authentication type AuthType Basic # The name shown in the browser's password dialog AuthName "Restricted Area — Authorised Users Only" # Point to the .htpasswd file (use an absolute path) AuthUserFile /etc/apache2/.htpasswd # Require a valid user from the .htpasswd file Require valid-user

IP-Based Access Control

.htaccess — Allow or Deny by IP Address
Restrict access to specific IP addresses or ranges. Useful for admin areas that should only be accessible from a known office IP.
# Allow only a specific IP — deny everyone else Require ip 192.168.1.100 # Allow a subnet (CIDR notation) Require ip 192.168.1.0/24 # Allow multiple IPs Require ip 192.168.1.100 203.0.113.50 # Allow a specific IP OR a valid password user (either is sufficient) <RequireAny> Require ip 192.168.1.0/24 Require valid-user </RequireAny> # Block a specific IP, allow everyone else <RequireAll> Require all granted Require not ip 203.0.113.99 </RequireAll>

Disable Directory Listing

.htaccess — Prevent Apache Listing Directory Contents
By default, if no index file is found, Apache may display a list of all files in the directory. This is a security risk — disable it.
# Remove the Indexes option — disables automatic directory listings Options -Indexes # Also deny direct access to sensitive file types <FilesMatch "\.(env|log|sql|bak|conf)$"> Require all denied </FilesMatch>
⚠️ HTTP Basic Auth sends credentials Base64-encoded, not encrypted. Always combine Basic Authentication with HTTPS — over plain HTTP, usernames and passwords can be intercepted trivially. Basic Auth over HTTP is effectively no security at all.

6 — Rewrite a URL with a .htaccess File

mod_rewrite is Apache's URL rewriting engine. It allows you to transform request URLs on the fly — redirecting HTTP to HTTPS, removing www, creating clean URLs, or routing all requests through a single entry point (as used by most PHP and Python web frameworks).

Prerequisite: mod_rewrite must be enabled and AllowOverride All must be set for the directory in the Apache configuration. Enable on Debian with sudo a2enmod rewrite; on Red Hat it is usually enabled by default.

RewriteRule Syntax

Understanding the RewriteRule Format
# RewriteRule Pattern Substitution [Flags] # # Pattern — a regex matched against the URL path (after the domain) # Substitution — what to rewrite the URL to # Flags — modifiers in square brackets # Common flags: # [L] — Last rule: stop processing further rules if this matches # [R=301] — Redirect with HTTP 301 (permanent) # [R=302] — Redirect with HTTP 302 (temporary) # [NC] — No Case: case-insensitive pattern matching # [QSA] — Query String Append: keep original query string # [P] — Proxy: pass the request to a backend (requires mod_proxy) # [F] — Forbidden: return 403 for matching requests # [G] — Gone: return 410 for matching requests

Common Rewrite Recipes

Recipe 1 — Force HTTPS (redirect all HTTP traffic)
RewriteEngine On # Check if the connection is NOT already HTTPS RewriteCond %{HTTPS} off # Redirect everything to the HTTPS equivalent URL RewriteRule ^ https://%{HTTP_HOST}%{REQUEST_URI} [L,R=301]
Recipe 2 — Force www (or remove www)
RewriteEngine On # Redirect non-www to www RewriteCond %{HTTP_HOST} !^www\. [NC] RewriteRule ^ https://www.%{HTTP_HOST}%{REQUEST_URI} [L,R=301] # --- OR --- redirect www to non-www RewriteCond %{HTTP_HOST} ^www\.(.+)$ [NC] RewriteRule ^ https://%1%{REQUEST_URI} [L,R=301] # %1 captures the domain without www from the RewriteCond group
Recipe 3 — Clean URLs (remove .html extension)
RewriteEngine On # If the request is not for a real file or directory... RewriteCond %{REQUEST_FILENAME} !-f RewriteCond %{REQUEST_FILENAME} !-d # ...try adding .html and serve it silently (no redirect) RewriteRule ^([^\.]+)$ $1.html [NC,L] # Result: /about → serves /about.html (URL stays clean)
Recipe 4 — Single Entry Point (SPA / Framework routing)
RewriteEngine On RewriteBase / # Don't rewrite actual files (CSS, JS, images) RewriteCond %{REQUEST_FILENAME} !-f # Don't rewrite actual directories RewriteCond %{REQUEST_FILENAME} !-d # Route everything else through index.html (React / Vue / etc.) RewriteRule . /index.html [L]
Recipe 5 — Permanent URL redirect (moved page)
RewriteEngine On # Redirect an old page URL to a new one permanently RewriteRule ^old-page\.html$ /new-page [R=301,L] # Redirect an entire old directory to a new location RewriteRule ^blog/(.*)$ /articles/$1 [R=301,L] # Example: /blog/2024/post-1 → /articles/2024/post-1
💡 Test rewrites with curl, not your browser. Browsers aggressively cache 301 redirects. If a rewrite rule is not behaving as expected, always test with curl -v http://yoursite.com/path first — it shows the exact HTTP response headers including Location. Clear your browser cache or use a private window when testing redirects.
