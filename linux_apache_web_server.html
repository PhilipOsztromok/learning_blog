🌐 Serve Web Content with Apache

Apache HTTP Server is the world's most widely deployed web server, powering a significant portion of the internet since 1995. This lesson covers everything from installing and configuring Apache to deploying sites, managing logs, and using virtual hosts — across both Debian-based (Raspberry Pi / Ubuntu) and Red Hat-based (RHEL / Rocky / Fedora) systems.

Platform note: Apache is called apache2 on Debian-based systems and httpd on Red Hat-based systems. Commands, file paths, and service names differ between the two — platform-specific blocks are clearly labelled throughout this lesson.

1 — How a Web Server Works

A web server is software that listens for incoming network requests and responds with content — HTML pages, images, CSS files, or data from an application. Apache handles this using the HTTP protocol (Hypertext Transfer Protocol) on port 80, or HTTPS on port 443.

💻 Browser Client
→ HTTP Request
GET /index.html
🔀 Network TCP Port 80/443
→ Routed to server
IP address
🖥️ Apache Web Server
← HTTP Response
200 OK + content
📄 Content HTML / Files
The Request
HTTP Methods
A client sends an HTTP request with a method (GET, POST, PUT, DELETE), a path (/index.html), and headers (browser info, cookies, content type). Apache reads this and decides how to respond.
The Response
Status Codes
Apache responds with a status code (200 OK, 404 Not Found, 301 Redirect, 500 Server Error), response headers, and a body (the HTML page, file, or error message).
Static Content
Files served directly
Apache reads files from the document root directory and returns them as-is. HTML, CSS, JavaScript, and images are all static. No processing — just file delivery.
Dynamic Content
Applications and proxies
With modules like mod_proxy or mod_wsgi, Apache can pass requests to a backend application (Python/FastAPI, PHP, Node.js) and return that application's response to the client.

2 — Ensure Network Connectivity

Before installing Apache, confirm your server has a working network connection and that any firewall is configured to allow web traffic on ports 80 and 443.

Check Your IP Address

🐧 Both Platforms
# Show all network interfaces and IP addresses ip addr show # or the shorthand: ip a # Look for your ethernet (eth0, ens3) or WiFi (wlan0) interface # Example output: 2: eth0: <BROADCAST,MULTICAST,UP,LOWER_UP> inet 192.168.1.50/24 brd 192.168.1.255 scope global eth0 # Test internet connectivity ping -c 4 8.8.8.8

Configure the Firewall

🐧 Debian / Raspberry Pi — ufw
# Install ufw if not present sudo apt install ufw # Allow HTTP and HTTPS sudo ufw allow 80/tcp sudo ufw allow 443/tcp # or by service name: sudo ufw allow 'Apache Full' # Enable the firewall sudo ufw enable sudo ufw status
🎩 Red Hat — firewalld
# Allow HTTP and HTTPS permanently sudo firewall-cmd --permanent \ --add-service=http sudo firewall-cmd --permanent \ --add-service=https # Apply changes sudo firewall-cmd --reload # Verify sudo firewall-cmd --list-all

Check for Port Conflicts

🐧 Both Platforms
# Check what is listening on port 80 before installing Apache sudo ss -tlnp | grep ':80' # or sudo netstat -tlnp | grep ':80' # If nothing is returned, port 80 is free — you're good to proceed

4 — Explore the Configuration File

🐧 Debian — /etc/apache2/ structure
# Main config file /etc/apache2/apache2.conf # Included directories /etc/apache2/conf-available/ # optional config fragments /etc/apache2/conf-enabled/ # symlinks to active configs /etc/apache2/mods-available/ # all available modules /etc/apache2/mods-enabled/ # symlinks to active modules /etc/apache2/sites-available/ # virtual host definitions /etc/apache2/sites-enabled/ # symlinks to active sites /etc/apache2/ports.conf # port listening config # View the main config cat /etc/apache2/apache2.conf
🎩 Red Hat — /etc/httpd/ structure
# Main config file /etc/httpd/conf/httpd.conf # Included directories /etc/httpd/conf.d/ # drop-in config files (*.conf) /etc/httpd/conf.modules.d/ # module loading configs /etc/httpd/modules/ # compiled module files (.so) # Red Hat uses a single directory — no # available/enabled split like Debian # View the main config cat /etc/httpd/conf/httpd.conf

Key Configuration Directives

Core Directives Explained
# ServerRoot — base directory for Apache's own files ServerRoot "/etc/apache2" # Listen — which port(s) Apache accepts connections on Listen 80 Listen 443 # DocumentRoot — the directory from which files are served DocumentRoot "/var/www/html" # DirectoryIndex — default file when a directory is requested DirectoryIndex index.html index.php # ServerName — the server's hostname (add to avoid a warning) ServerName localhost # Directory block — set permissions and options for a path <Directory "/var/www/html"> Options Indexes FollowSymLinks AllowOverride All # allow .htaccess files Require all granted # allow access to everyone </Directory> # Include — pull in other config files Include conf-enabled/*.conf IncludeOptional sites-enabled/*.conf
.htaccess Files — Per-Directory Configuration
When AllowOverride All is set, Apache reads a .htaccess file in each directory, allowing per-site configuration without editing the main config. Commonly used for URL rewriting, authentication, and redirects.
# Example .htaccess for URL rewriting (requires mod_rewrite) Options -Indexes RewriteEngine On RewriteBase / # Redirect all HTTP traffic to HTTPS RewriteCond %{HTTPS} off RewriteRule ^ https://%{HTTP_HOST}%{REQUEST_URI} [L,R=301] # Serve index.html for all routes (SPA / React apps) RewriteCond %{REQUEST_FILENAME} !-f RewriteCond %{REQUEST_FILENAME} !-d RewriteRule . /index.html [L]

5 — Start Up and Shut Down the Web Service

Apache is managed via systemctl on all modern Linux distributions. The service name differs between platforms.

🐧 Debian / Raspberry Pi — apache2
# Start sudo systemctl start apache2 # Stop sudo systemctl stop apache2 # Restart (drops all connections) sudo systemctl restart apache2 # Reload config (graceful, no downtime) sudo systemctl reload apache2 # Check status sudo systemctl status apache2 # Enable at boot sudo systemctl enable apache2 # Disable at boot sudo systemctl disable apache2
🎩 Red Hat — httpd
# Start sudo systemctl start httpd # Stop sudo systemctl stop httpd # Restart (drops all connections) sudo systemctl restart httpd # Reload config (graceful, no downtime) sudo systemctl reload httpd # Check status sudo systemctl status httpd # Enable at boot sudo systemctl enable httpd # Disable at boot sudo systemctl disable httpd

Check Configuration Before Restarting

🐧 Debian
# Test syntax — always do this before restarting! sudo apache2ctl configtest # or sudo apache2 -t Syntax OK
🎩 Red Hat
# Test syntax sudo apachectl configtest # or sudo httpd -t Syntax OK
💡 Use reload, not restart, for config changes. systemctl reload sends a SIGHUP signal that causes Apache to re-read its configuration without closing existing connections. restart kills all connections. For a production server, always prefer reload when only the config has changed.

6 — Deploy a Site

Deploying a static site to Apache is a case of copying your files to the document root and ensuring the correct ownership and permissions.

Create and Deploy a Simple Site
# The default document root is /var/www/html on both platforms # Remove the default placeholder page sudo rm /var/www/html/index.html # Create a new homepage sudo nano /var/www/html/index.html
Example index.html
<!DOCTYPE html> <html lang="en"> <head> <meta charset="UTF-8"> <title>My Site</title> </head> <body> <h1>Hello from Apache!</h1> <p>My web server is working.</p> </body> </html>

File Permissions and Ownership

🐧 Debian / Pi — Apache user: www-data
# Set correct ownership sudo chown -R www-data:www-data /var/www/html # Set correct permissions # Directories: 755 (rwxr-xr-x) sudo find /var/www/html -type d \ -exec chmod 755 {} \; # Files: 644 (rw-r--r--) sudo find /var/www/html -type f \ -exec chmod 644 {} \;
🎩 Red Hat — Apache user: apache
# Set correct ownership sudo chown -R apache:apache /var/www/html # Set permissions (same as Debian) sudo find /var/www/html -type d \ -exec chmod 755 {} \; sudo find /var/www/html -type f \ -exec chmod 644 {} \; # SELinux: restore correct context sudo restorecon -Rv /var/www/html
On Red Hat with SELinux enforcing, always run restorecon after copying new files.

7 — Configure Logging

Apache writes two log files by default: the access log (every request) and the error log (problems and diagnostic messages). These are your primary diagnostic tools.

🐧 Debian — Log Locations
/var/log/apache2/access.log # all requests /var/log/apache2/error.log # errors & warnings # Watch the access log in real time sudo tail -f /var/log/apache2/access.log # Watch the error log sudo tail -f /var/log/apache2/error.log
🎩 Red Hat — Log Locations
/var/log/httpd/access_log # all requests /var/log/httpd/error_log # errors & warnings # Watch the access log in real time sudo tail -f /var/log/httpd/access_log # Watch the error log sudo tail -f /var/log/httpd/error_log
Log Format and LogLevel Directives
# In the Apache config (apache2.conf or httpd.conf) # LogLevel controls verbosity (least → most detailed) # emerg, alert, crit, error, warn, notice, info, debug LogLevel warn # default — suitable for production LogLevel debug # maximum detail — use only for troubleshooting # LogFormat defines what each access log entry contains # This is the "Combined Log Format" — the industry standard LogFormat "%h %l %u %t \"%r\" %>s %O \"%{Referer}i\" \"%{User-Agent}i\"" combined # CustomLog — which log file and which format to use CustomLog ${APACHE_LOG_DIR}/access.log combined # ErrorLog — where to write error messages ErrorLog ${APACHE_LOG_DIR}/error.log # Example access log entry explained: # 192.168.1.5 - - [15/Aug/2025:10:22:01 +0000] "GET /index.html HTTP/1.1" 200 648 # [client IP] [ident] [auth user] [time] [request] [status] [bytes]
Log Rotation: Apache logs are automatically rotated by logrotate on both platforms. The configuration lives in /etc/logrotate.d/apache2 (Debian) or /etc/logrotate.d/httpd (Red Hat). By default, logs are rotated weekly, compressed, and kept for 52 weeks. You can adjust this without any manual intervention needed for most deployments.

8 — Add Modules to Extend Apache

Apache's functionality is extended through modules. Essential modules for web development include mod_rewrite (URL rewriting), mod_ssl (HTTPS), and mod_proxy (reverse proxy to backend apps).

🐧 Debian — a2enmod / a2dismod
# Enable a module sudo a2enmod rewrite sudo a2enmod ssl sudo a2enmod proxy sudo a2enmod proxy_http sudo a2enmod headers # Disable a module sudo a2dismod rewrite # List enabled modules apache2ctl -M # Reload after enabling/disabling sudo systemctl reload apache2
🎩 Red Hat — LoadModule in config
# Most modules are pre-configured in: /etc/httpd/conf.modules.d/ # Install additional modules sudo dnf install mod_ssl sudo dnf install mod_security # LoadModule directive in httpd.conf LoadModule rewrite_module modules/mod_rewrite.so # List loaded modules httpd -M # Reload after changes sudo systemctl reload httpd

9 — Use Virtual Hosts to Serve Many Sites

Virtual hosting allows a single Apache server to serve multiple websites — distinguishing them by domain name (name-based virtual hosting). Each site gets its own document root, log files, and configuration.

🐧 Debian — sites-available / sites-enabled
# Create a config file for the new site sudo nano /etc/apache2/sites-available/mysite.conf
# Virtual host config (mysite.conf) <VirtualHost *:80> ServerName www.mysite.com ServerAlias mysite.com DocumentRoot /var/www/mysite <Directory /var/www/mysite> AllowOverride All Require all granted </Directory> ErrorLog ${APACHE_LOG_DIR}/mysite-error.log CustomLog ${APACHE_LOG_DIR}/mysite-access.log combined </VirtualHost>
# Create the document root sudo mkdir -p /var/www/mysite sudo chown -R www-data:www-data /var/www/mysite # Enable the site (creates symlink in sites-enabled) sudo a2ensite mysite.conf # Disable a site sudo a2dissite 000-default.conf # Test config then reload sudo apache2ctl configtest sudo systemctl reload apache2
🎩 Red Hat — conf.d directory
# Drop config files directly into conf.d/ sudo nano /etc/httpd/conf.d/mysite.conf
# Virtual host config (mysite.conf) <VirtualHost *:80> ServerName www.mysite.com ServerAlias mysite.com DocumentRoot /var/www/mysite <Directory /var/www/mysite> AllowOverride All Require all granted </Directory> ErrorLog /var/log/httpd/mysite-error.log CustomLog /var/log/httpd/mysite-access.log combined </VirtualHost>
# Create the document root sudo mkdir -p /var/www/mysite sudo chown -R apache:apache /var/www/mysite # SELinux: set correct context sudo restorecon -Rv /var/www/mysite # Test config then reload sudo apachectl configtest sudo systemctl reload httpd
Reverse Proxy Virtual Host — Apache in front of FastAPI
A common pattern: Apache receives requests on port 80/443 and forwards them to your Python/FastAPI application running on port 8000. Enable mod_proxy and mod_proxy_http first.
<VirtualHost *:80> ServerName api.mysite.com ProxyPreserveHost On ProxyPass / http://127.0.0.1:8000/ ProxyPassReverse / http://127.0.0.1:8000/ # Pass real client IP to FastAPI RequestHeader set X-Forwarded-For %{REMOTE_ADDR}s ErrorLog /var/log/apache2/api-error.log CustomLog /var/log/apache2/api-access.log combined </VirtualHost> # Enable required modules (Debian): # sudo a2enmod proxy proxy_http headers # sudo systemctl reload apache2

10 — Common Troubleshooting Topics

🔴 Apache won't start
Check config syntax first: apache2ctl -t (Debian) or httpd -t (RHEL). Then check error log: sudo journalctl -u apache2 -n 50. Most startup failures are config syntax errors or port conflicts.
🔴 Port 80 already in use
Run sudo ss -tlnp | grep :80 to find what's using the port. Common culprits: nginx, another Apache instance, or a Python dev server. Stop the conflicting service before starting Apache.
🔴 403 Forbidden error
Usually a permissions problem. Check: (1) file permissions (ls -la /var/www/html), (2) directory ownership matches Apache user (www-data or apache), (3) Require all granted is in the Directory block, (4) on RHEL, run restorecon -Rv /var/www/html for SELinux.
🔴 404 Not Found
Check the DocumentRoot path is correct and the file actually exists there. Verify the virtual host is enabled. On Debian, confirm the site symlink exists in sites-enabled/. Check if mod_rewrite is redirecting the request incorrectly.
🔴 Changes not taking effect
Run sudo systemctl reload apache2 (or httpd) after every config change. Browser cache can also hide changes — try a hard refresh (Ctrl+Shift+R) or test with curl -v http://localhost which bypasses the browser cache entirely.
🔴 SELinux blocking Apache (Red Hat)
Check the audit log: sudo ausearch -c httpd --raw | audit2allow. Quick test — temporarily set SELinux permissive: sudo setenforce 0. If site works, SELinux is the cause. Fix with: sudo restorecon -Rv /var/www/html or generate a custom policy.
🔴 Cannot connect from another machine
Check the firewall: sudo ufw status (Debian) or sudo firewall-cmd --list-all (RHEL). Verify Apache is listening on 0.0.0.0:80 not 127.0.0.1:80 using sudo ss -tlnp | grep :80. Check the Listen directive in ports.conf or httpd.conf.
🔴 .htaccess not working
Ensure AllowOverride All (not None) is set for the directory in the Apache config. Ensure mod_rewrite is enabled. Verify the .htaccess file has correct permissions (644) and is in the correct directory.
Essential Diagnostic Commands — Quick Reference
# Check service status sudo systemctl status apache2 # Debian sudo systemctl status httpd # Red Hat # Check config syntax sudo apache2ctl configtest # Debian sudo apachectl configtest # Red Hat # View recent log entries sudo tail -n 50 /var/log/apache2/error.log sudo tail -n 50 /var/log/httpd/error_log # View systemd journal for Apache sudo journalctl -u apache2 -n 100 --no-pager # Test the server locally curl -v http://localhost curl -I http://localhost # headers only # Check which ports Apache is listening on sudo ss -tlnp | grep httpd sudo ss -tlnp | grep apache # List enabled modules apache2ctl -M 2>/dev/null # Debian httpd -M 2>/dev/null # Red Hat # List enabled sites (Debian only) ls -la /etc/apache2/sites-enabled/
